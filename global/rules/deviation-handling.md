# Deviation Handling — 偏差处理规则

## 来源

提炼自 [get-shit-done](https://github.com/gsd-build/get-shit-done) 的 Deviation Handling 4 规则。

## 核心规则

实现过程中发现偏离原计划的情况时，按以下 4 条规则处理：

### 规则 1：自动修复 Bug

**条件**：发现已破坏的行为、运行时错误、安全漏洞
**动作**：直接修复，不需要确认
**示例**：发现 SQL 查询缺少参数绑定 → 直接加参数化查询

### 规则 2：自动补全缺失的关键功能

**条件**：发现缺少错误处理、输入验证、认证检查等关键路径缺失
**动作**：直接补全，不需要确认
**示例**：API 端点缺少权限检查 → 直接加 `@require_permission`

### 规则 3：自动修复阻塞问题

**条件**：缺少依赖、类型错误、import 失败等导致无法继续的问题
**动作**：直接修复，不需要确认
**示例**：新增的模块忘记 import → 直接补上

### 规则 4：架构变更必须确认

**条件**：涉及以下任何一项
- 修改数据库 schema（新增/删除/修改表或列）
- 改变 API 接口签名（参数/返回值/路由）
- 引入新的依赖或外部服务
- 改变模块间调用关系或数据流
- 修改认证/授权流程
- 影响多个文件（>5 个）的重构

**动作**：STOP → 报告变更内容和原因 → 等待用户确认后继续

```
偏差发现 → 判断类型
  ├── Bug/错误/漏洞        → 规则 1：直接修
  ├── 缺失关键功能         → 规则 2：直接补
  ├── 阻塞问题             → 规则 3：直接修
  └── 架构/schema 变更     → 规则 4：STOP + 确认
```

## 与其他规则的关系

| 规则 | 互补关系 |
|------|---------|
| `workflow.md` | workflow 管函数删除前的检查，本规则管实现中的偏差处理 |
| `testing.md` | testing 管测试规范，本规则管实现过程中发现的问题 |
| `database.md` | database 管 DB 操作规范，本规则的架构变更确认包含 schema 变更 |
