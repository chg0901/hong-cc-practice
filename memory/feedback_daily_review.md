**规则**：每次新会话开始时（用户发出第一条任务指令前），**必须先执行**以下步骤：

1. 读取最近 3 天的 `docs/work_summary_YYYYMMDD.md` 文件
2. 搜索所有 `- [ ]` 未完成待办项
3. 按优先级整理并报告给用户（P0 紧急 → P1 功能 → P2 优化 → P3 探索）
4. 询问用户：继续未完成项，还是开始新任务？

**Why**：2026-04-09~04-15 一周内 ~18 项待办分散在 7 个文件中，其中 04-11/13/14 三个文件甚至缺少"待办事项汇总"章节，导致待办断裂追踪失败。用户明确要求建立每日审查机制（2026-04-16）。

**How to apply**:
- 不要跳过这步直接开始新任务（除非用户明确说"忽略待办"）
- 04-08/09/10 的旧 UI 测试大多已被 04-15 的 Vision MCP 验证覆盖，可标记为已解决
- 功能待办（如项目复制去重、faults 表清理）如未处理则持续携带
- 与 `todo-tracking.md` 互补：todo-tracking 管格式，daily-review 管执行

## 待办优先级分类

| 级别 | 含义 | 示例 |
|------|------|------|
| P0 | 紧急 bug / 阻塞开发 | 登录失败、API 报错、数据丢失 |
| P1 | 本周应完成的功能/验证 | 截图补全、MCP 可用性验证 |
| P2 | 优化 / 数据清理 | 表去重、性能优化 |
| P3 | 探索性 / 评估 | 评估新 skill 效果、测试发布流程 |
