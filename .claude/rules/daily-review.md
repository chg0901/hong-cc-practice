# Daily Morning Review Rule

## Trigger

每次新会话开始时（用户发出第一条任务指令前），**必须先执行**待办审查。

## 检查流程（mandatory）

```
Step 1: 读取最近 3 天的 docs/work_summary_YYYYMMDD.md
Step 2: 提取所有 `- [ ]` 未完成项
Step 3: 按优先级分类报告（P0 → P1 → P2 → P3）
Step 4: 询问用户：继续未完成项，还是开始新任务？
```

**豁免条件**：
- 用户明确说"忽略待办"或"开始新任务"
- 用户报告紧急 bug（P0），直接处理
- 用户续接上一个会话的明确任务

## 优先级分类

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| P0 | 紧急 bug / 阻塞开发 | 立即修复，不询问 |
| P1 | 本周应完成的功能/验证 | 推荐优先处理 |
| P2 | 优化 / 数据清理 | 建议本周处理 |
| P3 | 探索性 / 评估 | 可延后，有闲暇时处理 |

## 与其他规则的关系

| 规则 | 管辖范围 | 互补关系 |
|------|----------|----------|
| `todo-tracking.md` | 会话内待办格式规范（markdown 结构） | daily-review 管执行，todo-tracking 管格式 |
| `work-summary-rules.md` | 文档结构和提交检查点 | work_summary 写完后，daily-review 负责下次读取 |
| `extension-onboarding.md` | 安装后整理流程 | onboarding 完成后产生的待办纳入 daily-review 追踪 |

## 会话结束时

在待办汇总中标注"下次会话优先处理"的项，使用 `**[下次优先]**` 标记：

```markdown
- [ ] **[下次优先]** 验证 Jina MCP 工具可用性
- [ ] 评估 /wiki-ingest 整理 docs/
```
