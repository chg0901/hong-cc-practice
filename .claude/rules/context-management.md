# Context Management Rules — 上下文管理与 Prompt Cache 策略

## 核心原则

**缓存还热、任务没换，继续聊；缓存过期、任务切换、噪音太多，果断重开。**

## 6 个核心指令速查

| 指令 | 用途 | 缓存影响 | 使用时机 |
|------|------|----------|----------|
| `/context` | 查看 token 占用分布 | 只读，不影响缓存 | 每个会话开头、装新 skill/MCP 后 |
| `/resume` | 恢复上次会话 | 命中缓存（1hr 内），省 90% | 任务没变、距离上次 <1hr |
| `/rewind` | 回滚到任意消息 | 保留回滚点之前的缓存 | 方案失败、上下文被污染 |
| `/clear` | 彻底重开 | **完全重建**，~50K token 全价 | 任务切换、缓存过期、噪音太多 |
| `/compact` | 有损压缩 | **当前缓存失效**，用总结替换 | 阶段性完成时主动触发 |
| subagent | 隔离脏活 | 独立窗口（5min 缓存），主会话不受影响 | 只要结论不要过程的工作 |

## Prompt Cache 三条规则

### 1. 前缀匹配

缓存只认从头开始、一字不差的前缀。第 1 页改一个字，整份缓存作废。

Claude Code 输入布局（不变 → 变化）：

```
系统提示 + 工具定义 + CLAUDE.md  →  对话历史  →  新消息
[不变，天然命中缓存]              [增长中]      [全价]
```

### 2. 缓存窗口

| 上下文 | 缓存寿命 | 每次命中刷新 |
|--------|---------|------------|
| 主会话 | 1 小时 | 是 |
| 子代理 | 5 分钟 | 是 |

闲置超过窗口时间 → 缓存过期 → 下次全价重建。

### 3. 成本差异

- 缓存命中：**1/10 价格**（90% 节省）
- 缓存未命中：全价
- 基础设施（系统提示 + 工具 + CLAUDE.md）：约 **50,000 token**，每次 /clear 都要全价重建

## "继续聊 vs 开新会话"决策表

### 继续当前会话（任一满足）

- 任务没变（还在调同一个 bug / 写同一个模块 / 围绕同一组文件）
- 距离上一条消息不到 1 小时（缓存还活着）
- 上下文里的内容还在有用（之前读过的文件、讨论过的方案还在用）

### 果断开新会话（任一满足）

- 任务换了（从认证做到支付，上下文完全不同）
- 闲置超过 1 小时（缓存大概率过期）
- 上下文被噪音塞满（试了十几种方案、读了一堆无关文件，输出质量下滑）

## 三条反直觉规则

### 1. 频繁 /clear 不是省钱，是烧钱

每次新会话重建 ~50K token 基础设施全价。活跃会话中这 50K 一直躺在缓存里，只付 1/10。

**规则**：做完一个小任务不 /clear，只有满足上方"开新会话"条件时才 /clear。

### 2. 一次做对比来回改三轮便宜

复杂任务开着 thinking 一次搞定，比关掉 thinking 来回改三轮更省。每多一轮对话，整个上下文重新传输一次。

**规则**：简单任务用 `/effort low` 省钱；复杂重构保持 thinking 开启。

**Why**: 思考是一次性开销，上下文传输是每轮都要付的开销。

### 3. 给路径不贴内容

不要把一万行日志/代码粘贴进对话。给文件路径，让 Claude 自己用 Read/grep 提取相关行。

**规则**：长内容（日志、数据文件、大段代码）只给路径，不粘贴。

**Why**: 最便宜的 token 是那些根本没进上下文的 token。

## /rewind 进阶用法

1. 方案 A 失败 → **不要** 直接说"换个方案"
2. 先让 Claude `summarize from here` 做自我总结
3. /rewind 回到"文件读完、但还没试方案 A"的那一刻
4. 带着总结作为新 prompt 的开头重新下指令

```
# 错误：直接纠正
"刚才不对，换 B 试试"          ← 方案 A 的脏上下文仍在

# 正确：summarize → rewind → 新 prompt
1. "summarize from here: 方案 A 失败，原因是 X"
2. /rewind 回到读文件后
3. "别用 A，foo module 没暴露那个接口，直接上 B"
```

## /compact 进阶用法

- 默认：让 Claude 自己总结，lossy 压缩
- 带参数：`/compact focus on auth refactor, drop test debugging`
- **主动 compact** > 被动自动压缩（自动压缩时 Claude 可能砍掉重要信息）
- 时机：阶段性完成、缓存还热时主动执行

## 模型切换注意

- 缓存按模型隔离：Opus 积累的缓存，切到 Sonnet 后全部失效
- 需要轻量模型时，用 subagent 而非切主模型
- 便宜模型（如 GLM/Kimi）混用时，/exit 切换配置后 --resume 可以复用上下文

## Context Decay Prevention — 上下文腐烂防护

### 什么是上下文腐烂

上下文腐烂（Context Decay）是指随着对话轮次增加，Claude 的输出质量逐渐下降的现象。原因：
- 早期对话中的错误方案、废弃代码、误导性信息残留在上下文中
- 上下文窗口被无关文件内容填满，有效信息的密度降低
- 上下文越长，Claude 越容易"遗忘"早期的关键约束

### 检测信号

| 信号 | 说明 |
|------|------|
| Claude 重复已纠正过的错误 | 上下文中早期错误信息仍占权重 |
| 输出变泛泛/缺乏针对性 | 有效信息被噪音稀释 |
| 开始忽略 CLAUDE.md 中的规则 | 上下文过长，规则权重下降 |
| 同样的修改需要 2-3 轮才对 | 上下文中有矛盾信息干扰 |

### 应对策略

| 场景 | 策略 | 命令 |
|------|------|------|
| 大范围重构（>10 文件） | 用 subagent 隔离，每个 subagent 只负责 3-5 文件 | `context: fork` skill 或 Agent tool |
| 方案失败需要换方向 | summarize → rewind → 新 prompt | `/rewind` |
| 阶段性完成 | 主动 compact，保留关键信息 | `/compact focus on 当前任务, drop 已完成部分` |
| 读取大量无关文件后 | 立即 compact 或 rewind | `/compact` 或 `/rewind` |
| 探索性实验 | 用 Spike Skill（subagent 隔离） | `/spike` |
| 累积 3+ 失败尝试 | 果断 /clear 重开，把结论带入新会话 | `/clear` |

### Subagent 隔离模式（推荐用于大重构）

```
主会话（Orchestrator）           子代理（Worker）
┌──────────────────────┐    ┌──────────────────────┐
│ 读 plan              │    │ 独立 200k context    │
│ 分配任务块           │───→│ 执行 3-5 文件修改     │
│ 收集结果             │←───│ 返回 diff + 验证结果  │
│ 整合 + commit        │    │ 自动销毁             │
└──────────────────────┘    └──────────────────────┘

主会话上下文保持 30-40% 使用率
子代理用完即弃，不污染主上下文
```

### 与 GSD Wave Execution 的对比

| 维度 | 本项目策略 | GSD Wave Execution |
|------|-----------|-------------------|
| 隔离方式 | subagent + context:fork | 独立 executor 进程 |
| 上下文管理 | 主会话 < 40% + 主动 compact | 每个 plan 用 fresh 200k |
| 适用场景 | 中等规模（3-5 文件/任务） | 大规模（10+ 文件/phase） |
| 工具 | superpowers + Agent tool | GSD planner + executor agents |

## Token 效率监控（来自 ECC + OMC ecomode）

### GLM-5.1 上下文约束

当前使用 GLM-5.1 模型，上下文约 **500K token**（Claude Code 标准 1M 的一半），需要更积极的上下文管理：

| 约束 | 影响 | 应对策略 |
|------|------|----------|
| 上下文减半 | 可容纳的对话轮次减半 | 每完成一个任务批次立即 `/compact` |
| 上下文减半 | 35 个 rules 加载占比更高 | 保持每个 rule 精简（<200 行） |
| 上下文减半 | 大文件读取更容易溢出 | 用 subagent 隔离大量文件读取 |

### 80% 阈值规则

当感觉上下文使用超过 80% 时，**主动**建议用户：
1. `/compact focus on 当前任务, drop 已完成部分`
2. 或 `/clear` 重开新会话

检测信号：
- 输出变泛泛、缺乏针对性
- 重复已纠正过的错误
- 遗漏早期对话中的关键约束
- 同样的修改需要 2-3 轮才对

### ecomode 策略（来自 OMC）

| 场景 | 策略 | 节省效果 |
|------|------|----------|
| 简单查询/格式化 | `/effort low` | 减少 thinking token |
| 大量文件调研 | subagent 隔离 | 主会话保持干净 |
| 长内容 | 给路径不贴内容 | 最便宜的 token 是没进上下文的 token |
| 同一会话 3+ 模块 | 建议用户 `/clear` | 避免上下文腐烂 |
| 阶段性完成 | 主动 `/compact` | 比被动自动压缩更可控 |

### Wave Execution（来自 GSD）

当任务涉及 **10+ 文件**时，使用 subagent 并行执行：
- 每个 subagent 负责 3-5 文件
- 用 TaskCreate 建立依赖关系（addBlockedBy）
- 先执行无依赖的子任务，再执行有依赖的
- 与 superpowers:dispatching-parallel-agents 配合使用

| 规模 | 策略 |
|------|------|
| 1-5 文件 | 主会话直接执行 |
| 5-10 文件 | 主会话执行，完成后主动 compact |
| 10+ 文件 | Wave Execution：subagent 并行，每个 3-5 文件 |
